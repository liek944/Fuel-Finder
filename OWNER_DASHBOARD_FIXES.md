# Owner Dashboard Fixes - October 25, 2024

## Issues Fixed

### 1. ✅ Total Actions Counter Incrementing on Every Refresh

**Problem:**
The "Total Actions" stat in the Overview tab was increasing by ~11 every time the dashboard was refreshed.

**Root Cause:**
The `owner_dashboard_stats` view was counting ALL activity logs including authentication events:
- `auth_success` - logged on every page load
- `auth_attempt` - logged on API key verification
- `view_dashboard` - logged when viewing dashboard

Each page refresh triggered multiple auth events, causing the counter to continuously increase.

**Solution:**
Updated the `owner_dashboard_stats` view to exclude authentication and view logs from the `total_actions` count:

```sql
COUNT(DISTINCT CASE 
  WHEN oal.action_type NOT IN ('auth_success', 'auth_attempt', 'view_dashboard') 
  THEN oal.id 
END) AS total_actions
```

Now `total_actions` only counts meaningful actions like:
- `verify_price` - Approving price reports
- `reject_price` - Rejecting price reports  
- `update_station` - Updating station details
- `create_station` - Creating new stations

### 2. ✅ Stations Tab Not Loading (500 Error)

**Problem:**
The Stations tab showed "No stations assigned yet" and console showed:
```
Failed to load resource: the server responded with a status of 500
Failed to fetch stations: 500
```

**Root Cause:**
The `getOwnerStations()` and `getOwnerStation()` functions in `ownerController.js` were using incorrect image table structure:

```sql
-- Wrong (using non-existent entity_id/entity_type pattern)
LEFT JOIN images img ON img.entity_id = s.id AND img.entity_type = 'station'
```

The actual database schema uses direct foreign keys:
```sql
CREATE TABLE images (
  id SERIAL PRIMARY KEY,
  station_id INTEGER REFERENCES stations(id),  -- Direct FK, not entity_id
  poi_id INTEGER REFERENCES pois(id),          -- Direct FK, not entity_id
  ...
);
```

**Solution:**
Updated both query functions to use the correct schema:

```sql
-- Correct
LEFT JOIN images img ON img.station_id = s.id
```

Also changed to return `filename` instead of `image_url` since URLs should be generated by the backend transformer.

### 3. ✅ Missing Reporter Name in Price Reports

**Problem:**
The Pending Reports tab was trying to display `reporter_name` but the field didn't exist in the database, causing display issues.

**Root Cause:**
The `fuel_price_reports` table schema had:
- `reporter_ip` - IP address
- `reporter_identifier` - Browser fingerprint

But NOT:
- `reporter_name` - Display name

The frontend (OwnerDashboard.tsx line 351) and backend were expecting this field.

**Solution:**
Added `reporter_name` column to `fuel_price_reports` table:

```sql
ALTER TABLE fuel_price_reports
ADD COLUMN IF NOT EXISTS reporter_name VARCHAR(255) DEFAULT 'Anonymous';

-- Update existing records
UPDATE fuel_price_reports
SET reporter_name = COALESCE(reporter_identifier, 'Anonymous')
WHERE reporter_name IS NULL OR reporter_name = '';
```

### 4. ✅ Login Subdomain Detection Issue

**Problem:**
Sometimes the login showed error:
```json
{"error":"Owner not found","message":"Subdomain 'fuelfinder' is not registered","subdomain":"fuelfinder"}
```

**Root Cause:**
This occurs when users try to access the owner portal from the main backend domain (`fuelfinder.duckdns.org`) instead of the owner-specific domain (`ifuel-dangay-portal.netlify.app`).

**Solution:**
This is working as intended - the error message correctly indicates that users must access the portal through their assigned subdomain. The frontend's `extractSubdomain()` function correctly detects Netlify deployments with `-portal` suffix and extracts the owner name.

**User Action Required:**
Always access the owner portal through: `https://ifuel-dangay-portal.netlify.app`
NOT through: `https://fuelfinder.duckdns.org`

### 5. ✅ CORS Error on Price Reports Endpoint

**Problem:**
```
Access to fetch at 'https://fuelfinder.duckdns.org/api/owner/price-reports/pending' from origin 
'https://ifuel-dangay-portal.netlify.app' has been blocked by CORS policy
```

**Solution:**
The CORS configuration in `app.js` already includes the necessary headers:
- `x-api-key`
- `x-owner-domain`

This error was likely transient or related to the other 500 errors that prevented proper response. With the schema fixes, this should resolve.

```sql

  id SERIAL,
  station_id INTEGER,
  fuel_type VARCHAR(50),
  price DECIMAL(10, 2),

  ...

-- owner_dashboard_stats (counted ALL logs)
COUNT(DISTINCT oal.id) AS total_actions

  id SERIAL,
  station_id INTEGER,

  ...

-- owner_dashboard_stats (excludes auth logs)
COUNT(DISTINCT CASE 
  WHEN oal.action_type NOT IN ('auth_success', 'auth_attempt', 'view_dashboard') 
  THEN oal.id 
END) AS total_actions
```

## Rollback Plan

If issues occur after deployment:

```sql
-- Rollback view to original (if needed)
CREATE OR REPLACE VIEW owner_dashboard_stats AS
SELECT 
    o.id AS owner_id,
    o.name AS owner_name,
    o.domain,
    COUNT(DISTINCT s.id) AS total_stations,
    COUNT(DISTINCT CASE WHEN fpr.is_verified = TRUE THEN fpr.id END) AS verified_reports,
    COUNT(DISTINCT CASE WHEN fpr.is_verified = FALSE THEN fpr.id END) AS pending_reports,
    COUNT(DISTINCT oal.id) AS total_actions,  -- Original (counts all logs)
    MAX(oal.created_at) AS last_activity
FROM owners o
LEFT JOIN stations s ON s.owner_id = o.id
LEFT JOIN fuel_price_reports fpr ON fpr.station_id = s.id
LEFT JOIN owner_activity_logs oal ON oal.owner_id = o.id
WHERE o.is_active = TRUE
GROUP BY o.id, o.name, o.domain;
```

The `reporter_name` column can remain as it's non-breaking (has DEFAULT value).
